# Этап 2. Проектный

**Тема ВКР:** Методика обнаружения несанкционированных действий и аномальной активности в корпоративной сети с использованием нейронных сетей.

**Цель этапа:** проектирование подсистемы предобработки сетевых потоков и схемы входных представлений, архитектуры нейросетевого детектора со схемой калибровки порога и формирования алертов, архитектуры программного агента на основе конечного автомата с библиотекой параметризованных шаблонов трафика и инжектором дрейфа распределений, формирование протокола экспериментального исследования с количественными критериями принятия.

**Покрываемые задачи ВКР:**

- задача 3 — проектирование методики и архитектуры детектора (детальное проектирование);
- задача 4 — разработка подсистемы предобработки и формирования обучающих и тестовых выборок (проектирование);
- задача 6 — разработка конечно-автоматного агента активного тестирования (проектирование);
- задача 8 — экспериментальная проверка эффективности методики (формирование протокола экспериментов).

**Связь с Этапом 1.** Все ключевые архитектурные решения опираются на результаты сравнительного анализа Этапа 1: целевая нейросетевая архитектура — гибрид **CNN-LSTM** (4,60 баллов из 5 по семи критериям, см. ## 1. Общая архитектура системы

Система проектируется в виде четырёх подсистем, согласованных с концептуальной моделью [Stage1 §6.1](../Stage1/report_stage1.md#61-концептуальная-модель-системы):

```
┌──────────────────┐    ┌──────────────────┐    ┌────────────────────┐    ┌─────────────────────┐
│  Источник данных │───►│   Предобработка  │───►│      Детектор      │───►│  Алертирование +    │
│  (CSV / NetFlow/ │    │  schema → clean  │    │   Conv1D → LSTM →  │    │  Drift-monitor      │
│   IPFIX / агент) │    │  encode → scale  │    │   FC → sigmoid     │    │  (severity / dedup) │
└──────────────────┘    │  → windowing     │    │  + калибровка τ    │    │                     │
        ▲               └──────────────────┘    └────────────────────┘    └─────────────────────┘
        │                                                                            │
        │                                                                            ▼
┌──────────────────┐                                                       ┌─────────────────────┐
│ FSM-агент актив- │                                                       │  Журналы / экспорт  │
│ ного тестирования│                                                       │  (JSON / REST API)  │
│ templates + FSM  │                                                       │                     │
│ + drift-injector │                                                       └─────────────────────┘
└──────────────────┘
```

Принципы проектирования:

1. **Конфигурируемость через YAML.** Каждый параметр любой подсистемы задаётся в декларативной форме (YAML), что обеспечивает воспроизводимость экспериментов (требование FR-10) [[1]](#ref-stage1).
2. **Единый формат представления.** Все подсистемы оперируют согласованной схемой потокового признака `FlowRecord`, что исключает test-time leakage и упрощает кросс-датасетные эксперименты.
3. **Слабая связность модулей.** Подсистема предобработки сериализуется в JSON и применяется идентично в обучении и инференсе.
4. **Открытая совместимость.** REST-интерфейс и формат `flow` совместимы с источниками NetFlow v9 (RFC 3954) [[2]](#ref-rfc3954) и IPFIX (RFC 7011) [[3]](#ref-rfc7011).

## 2. Проектирование подсистемы предобработки и формирования входных представлений

### 2.1. Схема признаков и валидация

Схема входных данных формализуется средствами Pydantic v2 (валидация типов и диапазонов на стадии загрузки). Для основного датасета UNSW-NB15 принят набор из 44 содержательных потоковых признаков, разделяемых на пять групп — согласно описанию датасета Moustafa и Slay (2015) [[4]](#ref-moustafa2015) и таксономии в [[5]](#ref-unsw-arxiv-2021).

**Таблица 2.1. Группы признаков UNSW-NB15.**

| Группа | Состав | Роль в детекции |
|---|---|---|
| Базовые потоковые | dur, proto, service, state, sbytes, dbytes, sttl, dttl, sloss, dloss, rate | длительность сессии, протокол, состояние сессии, байты и пакеты, общая скорость потока |
| Контентные | sload, dload, swin, dwin, smean, dmean, stcpb, dtcpb, trans_depth, response_body_len, spkts, dpkts | скорости, окна TCP, базовые транзакционные характеристики |
| Временные | sjit, djit, sinpkt, dinpkt, tcprtt, synack, ackdat | межпакетные интервалы, джиттер, RTT, timing-сигнатуры |
| Дополнительные флаги | ct_state_ttl, ct_flw_http_mthd, is_ftp_login, ct_ftp_cmd, is_sm_ips_ports | агрегаты по протокольному поведению, бинарные флаги |
| Контекстные ct_* | ct_srv_src, ct_srv_dst, ct_dst_ltm, ct_src_ltm, ct_src_dport_ltm, ct_dst_sport_ltm, ct_dst_src_ltm | агрегаты по последним 100 соединениям; отражают сканирование и фан-аут |

Дополнительно формируется унифицированный набор признаков, общих с CICIDS2017, на основе работы [[6]](#ref-advanced-ids-flow-2025) — для cross-evaluation в Этапе 4.

### 2.2. Очистка и обработка пропусков

Очистка проводится в следующей последовательности:

1. **Унификация регистра и пробелов** в категориальных признаках (`service`, `state`, `proto`) — устраняет варианты `tcp` / `TCP` / ` tcp`.
2. **Удаление дубликатов** — на стадии `fit` препроцессора. Для UNSW-NB15 типовое число дубликатов в train-set около 24,8 тыс. записей, что согласуется с очисткой, рекомендованной в обзорах датасетов [[7]](#ref-nids-datasets-review-2025).
3. **Заполнение пропусков:** для числовых — заменой на 0, для категориальных — константой `unknown` (frequency-encoding автоматически обрабатывает редкие/неизвестные категории).
4. **Обрезка крайних 0,1% хвостов** числовых распределений на уровне квантилей 0,001 и 0,999 — снижает влияние выбросов перед применением робастной нормализации [[8]](#ref-jier-preproc-2025).

**Важно.** Все статистики (медианы, IQR, квантили, частоты категорий) собираются *только* на обучающей выборке и затем применяются к валидационной, тестовой и стримовой. Это устраняет утечку информации (test-time leakage) и обеспечивает корректность оценки.

### 2.3. Кодирование категориальных признаков

В литературе по UNSW-NB15 применяются два подхода — Label Encoding и One-Hot [[8]](#ref-jier-preproc-2025), [[9]](#ref-jisem-unsw-2025). Label Encoding для номинальных категориальных вносит ложный порядок, поэтому в работе применяется комбинация:

- **One-Hot Encoding** для признаков с малой кардинальностью (`proto`, `state` — ≤ 12 уровней). Расширяет признаковое пространство на десятки бинарных столбцов;
- **Frequency Encoding** для признаков с большей кардинальностью (`service` — около 13 уровней, но может расти на других датасетах). Формула:

$$e_{\text{freq}}(c) = \frac{|\{i : x_i^{(c)} = c\}|}{n_{\text{train}}}$$

где $c$ — значение категориального признака, $n_{\text{train}}$ — размер обучающей выборки. Frequency Encoding сохраняет одно числовое значение на признак и устойчив к появлению новых категорий в эксплуатации (новой категории присваивается значение 0).

### 2.4. Нормализация числовых признаков

Числовые признаки в UNSW-NB15 имеют тяжёлые хвосты (степенное распределение байтов, скоростей, длительностей). Применяется двухступенчатая нормализация:

1. **Логарифмическое преобразование** $\tilde{x} = \log(1 + x)$ — сжимает диапазон, делает распределение ближе к нормальному.
2. **Robust Scaling** — нормализация по медиане и межквартильному размаху:

$$\tilde{x}' = \frac{\tilde{x} - \operatorname{med}(\tilde{x})}{\operatorname{IQR}(\tilde{x})}$$

Робастные оценки устойчивы к выбросам — критично для NIDS, где аномалии формируют «хвосты» распределения [[10]](#ref-scikit-robust).

Альтернативы:
- **Z-score / StandardScaler** — чувствителен к выбросам (среднее и СКО смещаются хвостами).
- **MinMax** — чувствителен к выбросам; новый максимум в эксплуатации ломает шкалирование.

Robust Scaling выбран как компромиссное решение, согласующееся с типовой практикой обработки тяжелохвостых распределений в NIDS [[8]](#ref-jier-preproc-2025).

### 2.5. Формирование скользящих окон

Архитектура CNN-LSTM требует на вход тензор $(B, W, F)$ — батч, длина окна, число признаков. Параметры скользящих окон:

| Параметр | Значение | Обоснование |
|---|:-:|---|
| Длина окна $W$ | 32 flow | Покрывает типовые DDoS-всплески и port-scan-сессии; согласуется с подходами 10–32 flow в современной литературе по flow-based NIDS [[11]](#ref-nids-sliding-window-2024), [[12]](#ref-multi-scale-nids-2026) |
| Шаг $S$ | 8 flow | 75 % перекрытие; повышает разрешение во времени без существенного роста объёма |
| Стратегия агрегации меток | last (метка последнего flow) | Эмпирически обоснованная структурой UNSW-NB15 — атаки формируют последовательности однотипных flow, и метка последнего в окне корректно отражает текущий режим |

Реализация — векторизованная (NumPy stride tricks), что обеспечивает построение окон без копирования данных на стадии стрима.

**Альтернативные стратегии агрегации меток** (рассматриваются в Этапе 4 как контрольные):
- `majority` — метка по большинству flow в окне;
- `any` — метка 1, если в окне хотя бы один атакующий flow;
- `consecutive` — учёт только подряд идущих атакующих flow.

Сравнительные эксперименты позволят оценить чувствительность F1 к стратегии агрегации.

### 2.6. Управление дисбалансом классов

Согласно [Stage1 §3.6](../Stage1/report_stage1.md#36-управление-дисбалансом-классов), применяется **focal loss** [[13]](#ref-focal-loss-2017):

$$\mathcal{L}_{\text{focal}} = -\frac{1}{n} \sum_{i=1}^{n} \alpha_{y_i} (1 - p_{y_i})^\gamma \log p_{y_i}$$

с параметрами по умолчанию $\alpha = 0{,}25$, $\gamma = 2{,}0$ (рекомендованы Lin и соавт. 2017). Дополнительно для классических baseline применяется class re-weighting:

$$w_c = \frac{n}{|\mathcal{C}| \cdot n_c}$$

где $n_c$ — число объектов класса $c$, $|\mathcal{C}|$ — число классов.

SMOTE и его варианты (Borderline-SMOTE, SMOTE-ENN, ADASYN) [[14]](#ref-smote-lstm-2023) не применяются в основном пайплайне по причинам: (а) синтез на уровне feature-векторов нарушает корреляционную структуру flow-признаков (особенно `ct_*`-агрегатов); (б) на UNSW-NB15 умеренный дисбаланс (доля атак около 55 % в train), focal loss обеспечивает достаточную компенсацию.

## 3. Проектирование архитектуры нейросетевого детектора

### 3.1. Pipeline инференса

Полный pipeline принимает на вход поток flow-записей и возвращает скоринг атаки и алерт:

```
flow → Schema → Cleanup → Encoding → Scaling → Windowing → Model → Threshold → Alert
```

Все стадии параметризуются YAML-конфигурацией и сериализуются в JSON для применения в эксплуатации.

### 3.2. Целевая архитектура CNN-LSTM

По результатам Этапа 1 целевой архитектурой принимается CNN-LSTM. Детальная схема:

```
Input(B, W=32, F=60)
   │
   ├─ Conv1D(F → 64, kernel=3, padding=same) → BN → ReLU → MaxPool(2) → Dropout(0,2)
   │
   ├─ Conv1D(64 → 128, kernel=3, padding=same) → BN → ReLU → MaxPool(2) → Dropout(0,2)
   │
   ├─ LSTM(input=128, hidden=64, num_layers=1, batch_first=True)
   │     └─ output: (B, W/4, 64) — берём последний скрытый h_T
   │
   ├─ FC(64 → 32) → ReLU → Dropout(0,3)
   │
   ├─ FC(32 → 1)
   │
   └─ Output: logit (B, 1) → Sigmoid → probability of attack
```

Архитектурные обоснования:

- **Два свёрточных блока с MaxPool(2)** — последовательное уменьшение временного разрешения с 32 до 8 шагов, при одновременном росте числа каналов с 60 до 128. Это обеспечивает иерархическое извлечение локальных паттернов (синдром SYN-флуда, периодичность сканирования и т.д.).
- **BatchNorm после Conv1D** — стабилизирует обучение, устраняет ковариационный сдвиг между батчами.
- **Один слой LSTM с hidden=64** — компромисс между ёмкостью и скоростью; согласуется с типовыми конфигурациями CNN-LSTM в литературе [[15]](#ref-cnnlstm-ieee-2024)–[[18]](#ref-cnnlstm-jcbi-2025).
- **Dropout 0,2/0,3** — стандартная регуляризация; величины подобраны по типовым значениям для CNN-LSTM в NIDS-задачах.
- **Sigmoid на выходе** — бинарная классификация (атака / норма); логиты сохраняются для калибровки.

Число обучаемых параметров — порядка 100 тыс., размер сохраняемой модели — ~0,7 МБ.

## 4. Калибровка вероятностей и порога принятия решения

### 4.1. Постановка задачи

Сырые выходные вероятности нейронных сетей систематически переуверены (overconfidence) — это документировано Guo и соавт. [[21]](#ref-temperature-scaling-2017) для широкого класса архитектур. Эксплуатационное применение в SOC требует, чтобы:

1. Вероятности соответствовали эмпирической частоте (низкий ECE — Expected Calibration Error).
2. Порог принятия решения соответствовал целевой эксплуатационной частоте ложных срабатываний (NFR-2: FPR ≤ 0,01).

### 4.2. Температурное масштабирование

Применяется однопараметрическое температурное масштабирование (Guo и соавт. 2017) [[21]](#ref-temperature-scaling-2017), [[22]](#ref-network-cal-2022):

$$\hat{p}_T(y = 1 \mid x) = \sigma\!\left(\frac{z(x)}{T}\right)$$

где $z(x)$ — логит модели, $T$ — параметр температуры, подбираемый на валидационной выборке. Оптимизация $T$ — через L-BFGS, минимизирующий negative log-likelihood на валидационных логитах:

$$T^* = \arg\min_T \, \operatorname{NLL}(\sigma(z / T), y_{\text{val}})$$

Температурное масштабирование выбрано как наиболее эффективный метод калибровки нейронных сетей с минимальным числом параметров (один скаляр $T$), не меняющий ранжирование логитов и, как следствие, не влияющий на ROC-AUC и PR-AUC [[21]](#ref-temperature-scaling-2017).

Качество калибровки оценивается через **Expected Calibration Error (ECE)** с 15 биннами равной ширины.

### 4.3. Поиск порога по target-FPR

Порог принятия решения $\tau$ калибруется под целевую частоту ложных срабатываний FPR* = 0,01 на валидационной выборке:

$$\tau^* = \min \{\tau : \operatorname{FPR}_{\text{val}}(\tau) \leq \operatorname{FPR}^*\}$$

Реализация — бинарный поиск по отсортированному списку скорингов нормальных окон. Двухступенчатая калибровка (сначала температурное масштабирование, затем порог) обеспечивает корректную интерпретируемость вероятностей и приемлемый эксплуатационный режим.

Альтернативно для отчётности фиксируется порог по F1-max:

$$\tau^{F1} = \arg\max_\tau \, F_1(\tau)$$

Сравнение режимов в работе SOC (target-FPR vs F1-max) выносится в Этап 4 (эксперимент E3).

## 5. Подсистема алертирования с уровнями критичности

### 5.1. Постановка задачи

Прямой выход детектора (бинарное решение «атака / норма») недостаточен для эксплуатации в SOC: ежедневный поток алертов среднестатистической организации достигает 4 500 событий, и 97 % аналитиков опасаются пропуска критического события [[23]](#ref-soc-alerts-2024). Архитектура алертирования должна обеспечить (а) ранжирование алертов по критичности, (б) подавление шумовых повторов через дедупликацию, (в) экспортный формат, совместимый с SIEM.

### 5.2. Severity ladder (пятиступенчатая шкала критичности)

Принимается пятиступенчатая шкала, согласованная с типовыми классификациями SIEM (CVSS-подобные уровни) [[24]](#ref-cvss-severity-2024):

| Уровень | Условие | Действие SOC |
|---|---|---|
| `info` | $s / \tau \in [1{,}0;\, 1{,}1)$ | информационное событие, без эскалации |
| `low` | $s / \tau \in [1{,}1;\, 1{,}3)$ | стандартный обзор аналитика L1 |
| `medium` | $s / \tau \in [1{,}3;\, 1{,}6)$ | приоритетный обзор аналитика L1, опционально передача L2 |
| `high` | $s / \tau \in [1{,}6;\, 2{,}0)$ | передача аналитику L2, расследование |
| `critical` | $s / \tau \geq 2{,}0$ | немедленное расследование, эскалация incident response |

где $s$ — скоринг детектора, $\tau$ — откалиброванный порог. Нормировка $s / \tau$ обеспечивает инвариантность severity к выбору $\tau$.

Альтернативные подходы:
- **Линейная шкала $s \in [\tau, 1]$ → severity** — даёт неравномерную плотность алертов в high/critical;
- **Энтропийная шкала** — требует калибровки распределения скорингов нормальных и атакующих окон.

Принятая нормировка $s / \tau$ выбрана как простая, интерпретируемая и устойчивая к смене порога.

### 5.3. Временная дедупликация

Атаки в потоке часто формируют последовательность подобных flow в течение секунд (DDoS-всплеск, port-scan), что приводит к серии срабатываний детектора на смежных окнах. Без дедупликации SOC получает «шторм алертов» по одному инциденту.

Реализуется **окно временной дедупликации** $\Delta t_{\text{dedup}} = 5$ с (типовая практика SIEM). В пределах окна алерты одной категории `<src_ip, dst_ip, attack_cat>` агрегируются в один совокупный алерт с обновлением максимальной severity и счётчиком повторов. Параметр $\Delta t_{\text{dedup}}$ настраивается через YAML.

### 5.4. Формат экспорта

Каждый алерт сериализуется в JSON Lines (`.jsonl`) для совместимости с SIEM:

```json
{
  "ts":             1716321712.345,
  "score":          0.873,
  "threshold":      0.675,
  "severity":       "high",
  "src_ip":         "10.0.0.42",
  "dst_ip":         "172.16.0.5",
  "attack_cat":     "Exploits",
  "dedup_count":    7,
  "model_version":  "cnn_lstm_seed42_v1",
  "ground_truth":   {"fsm_state": "ATTACK_EXPLOIT", "drift_type": null}
}
```

Поле `ground_truth` присутствует только при работе с FSM-агентом в активном тестировании и обеспечивает прямой расчёт per-state recall.

REST-эндпоинт `/alerts/recent` возвращает последние $n$ алертов.

## 6. Проектирование конечно-автоматного агента активного тестирования

### 6.1. Обоснование выбора подхода и связь с Этапом 1

В Этапе 1 ([Stage1 §5.3](../Stage1/report_stage1.md#53-конечно-автоматный-подход-к-шаблонной-генерации)) обоснован выбор гибридного подхода `templates + FSM + drift-injector` без cGAN-составляющей. Кратко: главные преимущества — прозрачная ground-truth, воспроизводимость, отсутствие коллапса режима (характерного для GAN [[25]](#ref-mode-collapse-2022)), интерпретируемость для аналитика SOC и согласованность шаблонов с таксономией UNSW-NB15.

Сходные подходы воспроизводимой шаблонной генерации трафика в литературе: ID2T [[26]](#ref-id2t-2020), ConCap [[27]](#ref-concap-2025), NetSecBed [[28]](#ref-netsecbed-2026), GothX [[29]](#ref-gothx-2024). Настоящая архитектура отличается интеграцией с инжектором дрейфа и наличием явной FSM-структуры.

### 6.2. Состояния FSM

Принимается 11 состояний автомата $\mathcal{S}$, согласованных с таксономией UNSW-NB15 и тремя типами дрейфа:

$$\mathcal{S} = \{\text{NORMAL}\} \cup \{\text{ATTACK}(c)\}_{c \in \mathcal{C}} \cup \{\text{DRIFT}(k)\}_{k \in \mathcal{K}}$$

где $\mathcal{C} = \{\text{DDoS}, \text{SCAN}, \text{BRUTE}, \text{EXPLOIT}, \text{EXFIL}, \text{INTREC}, \text{WORM}\}$ — 7 классов атак, $\mathcal{K} = \{\text{COV}, \text{PRIOR}, \text{CONCEPT}\}$ — 3 типа дрейфа.

**Таблица 6.1. Состояния FSM-агента.**

| Состояние | Тип | Шаблон трафика | Класс UNSW-NB15 |
|---|---|---|---|
| NORMAL | нормальный | normal | Normal |
| ATTACK_DDOS | атака | ddos_synflood | DoS |
| ATTACK_SCAN | атака | port_scan | Reconnaissance |
| ATTACK_BRUTE | атака | brute_force | Reconnaissance / Exploits |
| ATTACK_EXPLOIT | атака | exploit_payload | Exploits |
| ATTACK_EXFIL | атака | data_exfil | Backdoors |
| ATTACK_INTREC | атака | internal_recon | Reconnaissance |
| ATTACK_WORM | атака | worm_lateral | Worms |
| DRIFT_COV | дрейф | (модификация нормального трафика) | сдвиг масштаба/шума числовых признаков |
| DRIFT_PRIOR | дрейф | (изменение состава) | изменение долей классов в потоке |
| DRIFT_CONCEPT | дрейф | (модификация атак) | маскирование сигнатурных признаков |

### 6.3. Политика переходов

Политика $\pi : \mathcal{S} \to \Delta(\mathcal{S})$ задаётся стохастической матрицей вероятностей переходов в YAML-конфигурации `attacker/policy.yaml`. Дополнительно фиксируются **минимальные длительности** состояний — для каждой атаки задаётся минимум тиков, в течение которых агент остаётся в состоянии (предотвращает мгновенные «прыжки» через атакующее состояние без формирования характерных flow).

Пример фрагмента политики:

```yaml
transitions:
  NORMAL:
    NORMAL: 0.75
    ATTACK_DDOS: 0.04
    ATTACK_SCAN: 0.06
    ATTACK_BRUTE: 0.03
    ATTACK_EXPLOIT: 0.03
    ATTACK_EXFIL: 0.02
    ATTACK_INTREC: 0.04
    ATTACK_WORM: 0.01
    DRIFT_COV: 0.01
    DRIFT_PRIOR: 0.005
    DRIFT_CONCEPT: 0.005
  ATTACK_DDOS:
    NORMAL: 0.40
    ATTACK_DDOS: 0.60
  # ...
min_durations:
  ATTACK_DDOS: 8
  ATTACK_SCAN: 12
  ATTACK_BRUTE: 6
  # ...
```

Все строки матрицы переходов суммируются в 1,0 (инвариант, проверяется юнит-тестом). Минимальные длительности подобраны эмпирически из типичной длины соответствующих атак на UNSW-NB15.

### 6.4. Библиотека параметризованных шаблонов

Реализуются семь параметризованных шаблонов атак плюс шаблон `normal`. Каждый шаблон формирует последовательность объектов `FlowRecord` с явными значениями всех 44 признаков схемы UNSW-NB15.

**Таблица 6.2. Библиотека шаблонов атак.**

| Шаблон | Параметризация (YAML) | Класс UNSW-NB15 | Характерные признаки |
|---|---|---|---|
| `ddos_synflood` | интенсивность $\lambda_0$, длительность $\Delta$, доля SYN-флагов, распределение исходных IP | DoS | высокая скорость, доминирование SYN, малые `sbytes` |
| `port_scan` | скорость подключений $\rho$, диапазон портов, режим (SYN/CONNECT/FIN), длительность | Reconnaissance | малые dur, фан-аут на портах, малый `dbytes` |
| `brute_force` | интервал между попытками, целевой порт (SSH/FTP/HTTP), доля успехов | Reconnaissance / Exploits | повторяющиеся короткие сессии на одном порту |
| `exploit_payload` | размер payload, нестандартные сочетания TCP-флагов, доля retransmissions | Exploits | большие `sbytes`, аномальные `state` |
| `data_exfil` | длительность сессии, асимметрия $b_{\text{out}}/b_{\text{in}}$, стабильность скорости | Backdoors | длинные сессии, асимметричный объём, стабильная скорость |
| `internal_recon` | фан-аут потоков, периодичность, размер сессии | Reconnaissance | внутренние IP-источники, многочисленные dst |
| `worm_lateral` | число направленных потоков, периодичность, размер сессии | Worms | веерное распространение по портам и хостам |
| `normal` | смесь HTTP/HTTPS/DNS/SSH сессий с распределениями длительности и объёма по UNSW | Normal | согласовано с маргинальными распределениями UNSW NORMAL |

Параметры всех шаблонов задаются в `attacker/<template>.yaml`. Это обеспечивает:
- **Воспроизводимость** — побитовое воспроизведение прогона по seed и YAML;
- **Расширяемость** — добавление нового шаблона требует только нового YAML и регистрации;
- **Калибруемость** — параметры могут подгоняться под маргинальные распределения целевого датасета (KS-тест, MMD на режиме NORMAL).

### 6.5. Инжектор дрейфа распределений

Три типа дрейфа, согласованных с типологией Gama и соавт. [[30]](#ref-gama-drift-2014):

**1. Covariate drift (`DRIFT_COV`).** Изменение $P(X)$ при фиксированном $P(Y \mid X)$. Реализация: сдвиг масштаба и шума числовых признаков:

$$x'_i = (1 + \alpha \cdot u_i) \cdot x_i + \alpha \cdot \sigma_i \cdot \varepsilon_i$$

где $u_i \sim \mathcal{U}(-0{,}5; 0{,}5)$, $\varepsilon_i \sim \mathcal{N}(0, 1)$, $\sigma_i$ — стандартное отклонение признака на train, $\alpha$ — интенсивность дрейфа.

**2. Prior drift (`DRIFT_PRIOR`).** Изменение $P(Y)$ при сохранении $P(X \mid Y)$. Реализация: изменение долей классов в политике FSM.

**3. Concept drift (`DRIFT_CONCEPT`).** Изменение $P(Y \mid X)$. Реализация: маскирование сигнатурных признаков атак (зануление или подмена характерных значений `state`, `ct_*`-агрегатов) с сохранением остальных признаков на типичных значениях нормального трафика.

Контроль интенсивности — параметр $\alpha \in \{0; 0{,}25; 0{,}5; 0{,}75; 1{,}0\}$. Корректность задания интенсивности проверяется через статистические индексы PSI и MMD относительно эталонного окна нормы.

### 6.6. Runtime и ground-truth

Модуль `attacker/runtime.py` объединяет FSM-агент и drift-инжектор и предоставляет два режима работы:

- **offline** (`collect_dataframe`) — генерация фиксированного датафрейма для офлайн-экспериментов (E4, E5);
- **online** (`stream_with_callback`) — потоковая генерация с эмиссией `AgentTick` в детектор в реальном времени (E4, демо-прогон).

Каждый тик возвращает структуру:

```python
@dataclass
class AgentTick:
    timestamp:   float
    flow:        FlowRecord
    fsm_state:   str          # NORMAL / ATTACK_* / DRIFT_*
    attack_cat:  str | None   # None для NORMAL и DRIFT
    drift_type:  str | None   # COV / PRIOR / CONCEPT или None
    intensity:   float
```

Поле `ground_truth = (fsm_state, attack_cat, drift_type)` обеспечивает прямой расчёт **per-state recall** в Эксперименте E4 — аналитики, недоступной при использовании только статических датасетов. Этот аспект составляет существенную часть научной новизны работы (см. ВКР.txt, секция «Степень разработанности темы и научная новизна», п. 3).

## 7. Подсистема мониторинга дрейфа распределений

### 7.1. Состав индексов дрейфа

Реализуются три индекса, дополняющих друг друга методологически:

**Population Stability Index (PSI).** Покрытие маржинальных распределений, простая интерпретация, устоявшийся в финансовой и ML-аналитике инструмент [[31]](#ref-psi-fiddler-2024), [[32]](#ref-psi-arize-2024):

$$\operatorname{PSI} = \sum_{b} (p_b - q_b) \ln \frac{p_b}{q_b}$$

где $p_b$ — доля наблюдений в $b$-м бине эталонной выборки, $q_b$ — текущей. Принят стандартный порог $\operatorname{PSI} > 0{,}25$ — сигнал значимого дрейфа, требующего переобучения или адаптации [[31]](#ref-psi-fiddler-2024), [[33]](#ref-psi-radicalbit-2024).

**KL-расхождение Кульбака–Лейблера.** Дополняет PSI; чувствительнее к малым изменениям в хвостах распределений:

$$\operatorname{KL}(P \| Q) = \sum_b p_b \ln \frac{p_b}{q_b}$$

**Maximum Mean Discrepancy (MMD).** Покрывает многомерные совместные распределения; не требует биннинга. Используется характеристическое ядро — гауссово с автобандвидтом по медианной эвристике [[34]](#ref-mmd-median-2014):

$$\operatorname{MMD}^2(P, Q) = \mathbb{E}_{P,P}[k(x,x')] - 2\,\mathbb{E}_{P,Q}[k(x,y)] + \mathbb{E}_{Q,Q}[k(y,y')]$$

$$\sigma = \operatorname{median}\{\|x_i - x_j\|_2 : i \neq j\}$$

### 7.2. Логика срабатывания

Функция `drift_report` возвращает агрегированный отчёт по чанку:

```json
{
  "ts":          1716321712.345,
  "chunk_size":  300,
  "psi":         { "mean": 0.31, "max": 0.48, "n_features_over_threshold": 7 },
  "kl":          { "mean": 0.22, "max": 0.51 },
  "mmd":         { "value": 0.21, "bootstrap_p": 0.03 },
  "drift_alarm": true,
  "drift_type":  "covariate_suspected"
}
```

Индикатор `drift_alarm = true` устанавливается при выполнении хотя бы одного из условий:
- $\overline{\operatorname{PSI}} > 0{,}25$ по числовым признакам,
- $\operatorname{MMD}^2$ статистически значимо больше нулевой гипотезы по bootstrap-p ≤ 0,05.

### 7.3. Интеграция с инжектором дрейфа

Подсистема мониторинга работает независимо от FSM-агента и применима как к синтетическому, так и к реальному потоку. В Эксперименте E5 (см. §8.5) drift-monitor оценивается на стримах от FSM-агента с управляемой интенсивностью дрейфа $\alpha \in \{0; 0{,}25; 0{,}5; 0{,}75; 1{,}0\}$ — проверяется монотонность отклика индексов и корректность срабатывания `drift_alarm`.

## 8. Протокол экспериментального исследования

### 8.1. Общие принципы

Протокол построен по принципу «один эксперимент — один проверяемый вопрос». Каждый эксперимент имеет:

- **цель** — проверяемое утверждение;
- **условия** — конфигурация (модели, датасет, режим);
- **метрики** — количественные показатели;
- **критерий принятия** — численный порог, при выполнении которого утверждение считается подтверждённым.

Общие условия для всех экспериментов:
- одинаковый препроцессор и схема разделения train / val / test;
- порог принятия решения калибруется на val под целевой FPR = 0,01 (см. §4.3);
- статистическая оценка — bootstrap CI 95 %, парный критерий Уилкоксона при сравнении пар моделей;
- для ведущих моделей — $n \geq 5$ независимых seed-ов;
- все артефакты (метрики, конфиги, графики) сохраняются в `experiments/runs/`.

### 8.2. Эксперимент E1: сравнение моделей-кандидатов

**Цель.** Установить относительную эффективность 14 моделей-кандидатов на основном датасете в идентичных условиях обучения и оценки.

**Условия.** Один и тот же `Preprocessor` (sklearn-совместимый), `WindowBuilder($W=32, S=8$)`, одни и те же гиперпараметры обучения (`configs/train/full.yaml`). Для CNN-LSTM (proposed) и табличных baseline — по 3 seed-а (42, 123, 2024) с переходом на 5 seed-ов для финальной верификации; для остальных DL-моделей — по 1 seed = 42 в рамках вычислительного бюджета CPU.

**Метрики.** Precision, Recall, F1, MCC, ROC-AUC, PR-AUC; стандартное отклонение F1 по seed-ам.

**Критерий принятия.** Целевая модель (по результатам Этапа 1 — CNN-LSTM) обеспечивает F1 ≥ 0,90 на UNSW-NB15 (NFR-1a) и PR-AUC ≥ 0,95 (NFR-1b); статистически значимое превосходство ($p < 0{,}05$, парный Уилкоксон) либо равенство с лидером по F1 при значимо лучшей задержке инференса.

### 8.3. Эксперимент E2: анализ ошибок целевой модели

**Цель.** Идентификация классов атак с пониженной полнотой; выявление систематических ошибок FP/FN.

**Условия.** Целевая модель из E1 с откалиброванным порогом target-FPR = 0,01. Разбор на тестовой выборке.

**Метрики.** Матрица ошибок 2×2; per-class recall по 10 значениям `attack_cat` UNSW-NB15; ROC-кривые по классам; распределение FP/FN по контекстным признакам (`proto`, `service`, длительность сессии).

**Критерий принятия.** Идентифицированы классы атак с пониженной полнотой; полученный анализ ошибок используется для калибровки шаблонов в Эксперименте E4.

### 8.4. Эксперимент E3: калибровка вероятностей и порога

**Цель.** Проверка работоспособности процедуры температурного масштабирования и калибровки порога под target-FPR.

**Условия.** Целевая модель и совокупность DL-альтернатив.

**Метрики.** ECE до и после температурного масштабирования (15 биннов); температура $T^*$; калиброванный порог $\tau^*$; FPR на test после калибровки; FP-per-time, TTD.

**Критерий принятия.** Снижение ECE как минимум на 30 % относительно raw-режима; фактический FPR на test в пределах $0{,}01 \pm 0{,}005$.

### 8.5. Эксперимент E4: стресс-тест с FSM-агентом

**Цель.** Проверка устойчивости целевой модели в потоковом сценарии под управляемой нагрузкой FSM-агента; измерение per-state recall, недоступного на статических датасетах.

**Условия.** FSM-агент запускается в режиме `templates + FSM + drift` на 600 тиков, в потоковом режиме через `Preprocessor` и `WindowScorer`. Целевая модель из E1.

**Метрики.** Precision, Recall, F1, FPR, MCC по всему прогону; per-state recall и per-state FPR по 11 состояниям FSM; TTD по подмножеству ATTACK-состояний; FP-per-time на NORMAL-участках.

**Критерий принятия.** На NORMAL-участках FPR ≤ 0,01 (согласовано с E3); per-state recall ≥ 0,50 хотя бы для 5 из 7 классов атак.

### 8.6. Эксперимент E5: устойчивость к контролируемому дрейфу

**Цель.** Оценка чувствительности целевой модели и подсистемы мониторинга дрейфа к контролируемому дрейфу распределений.

**Условия.** Инжектор дрейфа применяется к стримам FSM-агента с интенсивностью $\alpha \in \{0; 0{,}25; 0{,}5; 0{,}75; 1{,}0\}$ для двух типов дрейфа: covariate и concept. Эталонное окно — 5 000 нормальных flow из train.

**Метрики.** F1, PR-AUC в зависимости от $\alpha$; индексы PSI, KL, MMD; индикатор `drift_alarm`; время восстановления F1 после возврата $\alpha \to 0$.

**Критерий принятия.**
- (а) ΔF1 ≤ 0,10 при PSI ≤ 0,25 (NFR-6);
- (б) подсистема мониторинга дрейфа корректно срабатывает (`drift_alarm = true`) при PSI > 0,25 во всех точках.

### 8.7. Эксперимент E6: эксплуатационная производительность

**Цель.** Оценка соответствия эксплуатационным требованиям NFR-3 (latency) и NFR-4 (throughput).

**Условия.** Все модели, CPU, batch = 1 для latency, batch = 256 для throughput. 50 прогонов с предварительным прогревом, медианное значение.

**Метрики.** Inference latency на одно окно, мс; throughput, EPS (events per second); пиковое потребление RAM.

**Критерий принятия.** Целевая модель: latency ≤ 50 мс/окно (NFR-3); throughput ≥ 1 000 EPS (NFR-4).

### 8.8. Демонстрационный прогон

**Цель.** Верификация работоспособности связки «FSM-агент → Preprocessor → Detector → AlertFormer → Drift-monitor» в потоковом режиме реального времени.

**Условия.** Прогон 300 тиков; FSM-агент выпускает смесь NORMAL и ATTACK состояний; алерты дедуплицируются с $\Delta t_{\text{dedup}} = 2$ с; drift-monitor работает с шагом 30 с.

**Артефакты.** `demo_alerts.jsonl` (с severity, score, ground-truth), `demo_drift.jsonl` (PSI, MMD по чанкам), `demo_summary.json`, `demo_timeline.pdf/png`.

**Критерий принятия.** Все компоненты pipeline отработали без сбоев; сформированы все артефакты в ожидаемых форматах.

## 9. Выводы по этапу

1. **Архитектура системы.** Спроектирована четырёхподсистемная архитектура — предобработка, детектор, активное тестирование, алертирование с мониторингом дрейфа. Принципы построения: декларативная YAML-конфигурируемость, слабая связность модулей, идентичность преобразований в обучении и инференсе, совместимость с NetFlow v9 / IPFIX.

2. **Подсистема предобработки.** Спроектирован полный pipeline: валидация Pydantic-схемы → очистка с обработкой дубликатов и обрезкой квантилей 0,1 % / 99,9 % → кодирование (one-hot + frequency) → нормализация (log-преобразование + Robust Scaling) → формирование скользящих окон $W = 32$, $S = 8$ с last-стратегией агрегации меток. Все статистики собираются исключительно на train-выборке.

3. **Архитектура детектора.** Зафиксирована детальная схема целевой архитектуры CNN-LSTM: два свёрточных блока (Conv1D → BN → ReLU → MaxPool → Dropout) → LSTM(hidden=64) → MLP-head → Sigmoid; обучение AdamW + cosine annealing с focal loss, early stopping по val_pr_auc, gradient clipping. Реализуется регистр из 14 моделей-кандидатов для эксперимента в равных условиях.

4. **Калибровка вероятностей и порога.** Спроектирована двухступенчатая калибровка: температурное масштабирование логитов через L-BFGS на val + поиск порога по target-FPR = 0,01 через бинарный поиск. Метрика качества калибровки — ECE с 15 биннами.

5. **Подсистема алертирования.** Спроектирована пятиступенчатая шкала severity (info / low / medium / high / critical) на основе нормированного превышения скоринга над порогом, временная дедупликация $\Delta t = 5$ с, экспорт в JSON Lines для совместимости с SIEM.

6. **FSM-агент активного тестирования.** Зафиксирована архитектура из 11 состояний (NORMAL + 7 ATTACK + 3 DRIFT), вероятностной политики переходов с минимальными длительностями состояний, библиотеки из семи параметризованных шаблонов атак плюс шаблон normal, и инжектора дрейфа трёх типов (covariate, prior, concept) с управляемой интенсивностью $\alpha \in \{0; 0{,}25; 0{,}5; 0{,}75; 1{,}0\}$.

7. **Подсистема мониторинга дрейфа.** Зафиксирована тройка взаимодополняющих индексов PSI / KL / MMD; принят стандартный порог $\operatorname{PSI} > 0{,}25$ для срабатывания `drift_alarm`; для MMD применяется гауссово ядро с медианной эвристикой автобандвидта.

8. **Протокол экспериментов.** Сформирован протокол из шести основных экспериментов E1–E6 плюс демонстрационный прогон и cross-evaluation как вспомогательный. Каждый эксперимент имеет цель, метрики, количественные критерии принятия и связь с конкретными нефункциональными требованиями.

Все принятые проектные решения опираются на результаты сравнительного анализа Этапа 1 и подтверждённые литературные практики 2017–2025 годов в IEEE, Springer, Elsevier, Nature, ACM и MDPI. Результаты Этапа 2 обеспечивают полное основание для перехода к Этапу 3 (программная реализация).
